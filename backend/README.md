# Cafe Finder Backend

Minimal Node.js Express backend to demonstrate:

- Verifying Firebase ID tokens using `firebase-admin`.
- Admin-protected endpoints to promote users and create cafes.
- Optional Postgres (Cloud SQL) integration via `DATABASE_URL`.
- Dockerfile for Cloud Run deployment.

Quick start (local):

1. Install deps

```bash
cd backend
npm install
```

2. Provide credentials and DB connection (see `.env.example`). For local testing set `ADMIN_SECRET` to a value and use it in requests via header `x-admin-secret`.

3. Run

```bash
npm run dev
```

Endpoints:

- `GET /health` ŌĆō health check
- `POST /admin/promote` ŌĆō body `{ uid, role }`, requires admin auth
- `GET /cafes` ŌĆō list cafes from Postgres
- `POST /cafes` ŌĆō create cafe (admin only)

Deploy to Cloud Run:

1. Build and push container (example):

```bash
gcloud builds submit --tag gcr.io/PROJECT_ID/cafe-finder-backend
gcloud run deploy cafe-finder-backend --image gcr.io/PROJECT_ID/cafe-finder-backend --region=REGION --platform=managed --allow-unauthenticated --set-env-vars=DATABASE_URL=YOUR_DB_URL
```

Make sure the Cloud Run service account has access to the Cloud SQL instance or use Cloud SQL Auth Proxy.

Database migrations
-------------------

Run migrations locally against the database referenced by `DATABASE_URL`:

```bash
cd backend
npm run migrate
```

This executes `migrations.sql` using the `pg` driver. For production, run migrations as part of your deploy pipeline or use a dedicated migration tool.

CI / GitHub Actions
-------------------

A sample GitHub Actions workflow is included at `.github/workflows/cloud-run.yml`. It expects the following repository secrets:

- `GCP_SA_KEY` ŌĆö JSON service account key with permissions to Cloud Build & Cloud Run.
- `GCP_PROJECT` ŌĆö your GCP project id.
- `GCP_REGION` ŌĆö region for Cloud Run (e.g., `us-central1`).
- `DATABASE_URL` ŌĆö connection string for the Postgres instance.

Ensure secrets are set before enabling the workflow.

Docker Compose local dev
------------------------

We included a `docker-compose.yml` that offers two services:

- `cloudsql-proxy` ŌĆö runs the Cloud SQL Auth Proxy and exposes port 5432. Place your service account JSON at `backend/cloudsql/service-account.json` and set `CLOUDSQL_CONNECTION_NAME` before starting.
- `postgres-local` ŌĆö a local Postgres image for quick development, exposed on host port 5433.

To run the local Postgres and apply migrations:

```bash
cd backend
docker compose up -d postgres-local
DATABASE_URL=postgres://dev:dev@localhost:5433/cafefinder npm run migrate
```


Client integration notes
------------------------

1. Initialize Firebase in your web app (Flutter web or plain JS). For Flutter web, keep `firebase_options.dart` generated by the Firebase CLI and call `Firebase.initializeApp()` in `main.dart`.

2. When calling protected backend endpoints, obtain the current user's ID token and send it in the `Authorization` header as `Bearer <ID_TOKEN>`.

Example (web, modular JS / compat):

```js
// using Firebase compat auth (or adapt to modular with getAuth())
const idToken = await firebase.auth().currentUser.getIdToken();
const res = await fetch('https://your-backend.example.com/admin/promote', {
	method: 'POST',
	headers: { 'Content-Type': 'application/json', 'Authorization': `Bearer ${idToken}` },
	body: JSON.stringify({ uid: 'USER_UID', role: 'admin' })
});
```

3. For local testing you can also use `ADMIN_SECRET` by sending header `x-admin-secret: <ADMIN_SECRET>` when calling endpoints that accept this bypass (see `server.js`). Do NOT expose `ADMIN_SECRET` publicly in production.
